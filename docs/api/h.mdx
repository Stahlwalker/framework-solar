A compact array-based notation for building vnode trees. An alternative to `createElement` that reads closer to the structure it describes.

```js
import { h } from 'solarbuild'

h(['div', { class: 'card' },
  ['h2', {}, 'Title'],
  ['p', {}, 'Body text'],
])
```

Equivalent to:

```js
createElement('div', { class: 'card' },
  createElement('h2', {}, 'Title'),
  createElement('p', {}, 'Body text'),
)
```

## Syntax

```js
h([type, props?, ...children])
```

Each node is an array where:
- `type`: a tag name string (`'div'`) or a registered component name (`'Button'`)
- `props`: optional object of attributes or props
- `children`: remaining items, each a nested array or a primitive

Primitives (strings, numbers) pass through unchanged.

## Registry dispatch

If the type matches a registered component name, `h()` calls that component instead of `createElement`:

```js
registry.register(Button)

h(['Button', { label: 'Save', onClick: handleSave, variant: 'primary' }])
// equivalent to: Button({ label: 'Save', onClick: handleSave, variant: 'primary' })
```

Prop validation still runs. A wrong prop type throws a `ContractError`.


## Nested trees

```js
h(['div', { class: 'layout' },
  ['header', {},
    ['h1', {}, 'Solar'],
  ],
  ['main', {},
    ['Button', { label: 'Get started', onClick: () => {}, variant: 'primary' }],
  ],
])
```

## Children on registered components

When `h()` calls a registered component and there are child nodes, they're passed as a `children` prop, not spread as arguments. Declare `children` in the schema to receive them:

```js
const List = defineComponent({
  name: 'List',
  props: {
    children: { type: 'array' },
  },
  render({ children }) {
    return createElement('ul', {}, ...children)
  },
})

h(['List', {},
  ['li', {}, 'Item 1'],
  ['li', {}, 'Item 2'],
])
// children → [vnode, vnode] passed as a prop
```

If you omit `children` from the schema, Solar emits a dev warning but does not throw. The prop is silently dropped from the resolved props object.

For plain DOM elements (`div`, `ul`, etc.), children are passed as normal arguments to `createElement`. No prop involved.

## Special props

### style

Pass `style` as an object of camelCased CSS properties. This form merges cleanly on re-renders:

```js
h(['div', { style: { backgroundColor: '#09080a', padding: '2rem' } }])
```

String styles also work and are applied via `setAttribute`:

```js
h(['div', { style: 'background: #09080a; padding: 2rem' }])
```

Both forms produce correct output. Use the object form when styles are dynamic or component-defined; use the string form for one-off static styles.

---

### innerHTML

Set raw HTML on a DOM node by passing an `innerHTML` prop. Children are ignored when this prop is present.

```js
h(['div', { class: 'highlight', innerHTML: '<span class="token keyword">const</span> x = 1' }])
```

Use this for pre-rendered or syntax-highlighted HTML, like output from Prism.js or a markdown renderer. Only use it with trusted content.

---

## When to use h() vs createElement

`h()` is more compact and closer to markup in structure. `createElement` is more explicit and easier to trace in a debugger. Both produce identical vnodes, so mix them freely.

`h()` is particularly well-suited as a generation target: the array format is easy for a model to produce without needing to know function call syntax.
